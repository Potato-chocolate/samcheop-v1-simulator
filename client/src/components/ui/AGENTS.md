# ui

## Purpose
shadcn/ui primitive library. Each file is a thin React wrapper around a Radix UI primitive (or a hand-rolled headless component) styled with Tailwind v4 and composed via the `cn()` helper. These are the building blocks the application uses for buttons, dialogs, inputs, tables, navigation, charts, sidebars, etc. The shadcn registry config at `../../../../components.json` (repo root) tracks which components are installed and how new ones get scaffolded.

## For AI Agents

### Working In This Directory
- **These files were originally generated by the shadcn CLI. They are still hand-edited freely (no upstream sync), but the conventions (variant maps via `cva`, ref-forwarding, Radix `Primitive.*` slots) must be preserved so the next `npx shadcn add` doesn't conflict.**
- To add a new primitive prefer: `pnpm dlx shadcn@latest add <component>` (uses `components.json` at the repo root). Don't hand-roll a primitive that's already in the registry.
- All components compose with `cn(...)` from `@/lib/utils`. Use it whenever you accept a `className` prop so callers can override styles.
- The simulator (Home.tsx) builds most of its visual surface with raw Tailwind classes rather than these primitives, but `Dialog`, `Button`, `Input`, and `Sonner` toaster are wired in via `App.tsx` / save-report dialog. When extending Home, prefer reaching for a primitive here before introducing new bespoke markup.
- Several primitives mount portals (`Dialog`, `DropdownMenu`, `Popover`, etc.). They render into `document.body` by default; do not assume DOM hierarchy when wiring event handlers.

### Testing Requirements
- No unit tests in this folder. Behavior is verified visually and through the integration in `pages/Home.tsx` / `ComponentShowcase.tsx`.

### Common Patterns
- Variants via `class-variance-authority` (`cva`) for size/intent/state.
- Named exports per component (no default exports).
- Files keep Radix-style display names (`Component.displayName = "Component"`) so React DevTools is usable.
